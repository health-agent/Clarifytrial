# 지원서 초안 — ClarifyTrial Agent

_현재 저장소(v1.2-final 연구 프로토타입, 테스트 102개 통과)를 근거로 작성한 신청서용 초안입니다._

## 1. 연구 목표 및 접근 방법

- **목표**: 환자 요약문과 여러 임상시험의 선정/제외 기준을 대조하여, 부족한 정보를 능동적으로 되묻고 근거와 함께 시험별 추천을 제시하는 **대화형 임상시험 추천 멀티에이전트 시스템**(ClarifyTrial Agent) 개발.
- **설계**: 10개 LLM 에이전트(기준 파싱, 환자 프로파일 이해·답변 정규화, 상태 추적, 근거 구성, 기준 매칭, 결측정보 탐지, 질문 생성, 답변 반영·부분 재평가, 시험 추천, 결과 설명)가 **중앙 공유 상태인 Eligibility State Tracker**(환자 세션 단위, patient_id 기준)를 통해 협업.
- **매칭·추천 전략**: LLM은 언어 이해(파싱·추출·질문·설명)만 담당하고, 적격성 판정은 **결정론적 규칙 계층**(rules.py)이 전담 — 기준 유형×매칭 상태의 고정 매핑, 우선순위 규칙(차단 기준 → 부적격, 검토 필요 → 인간 검토, 높은 불확실성 → 불확실, 그 외 → 적격 가능), 관련도 점수는 **랭킹에만** 반영.
- **단순 챗봇이 아닌 이유**: (1) 시험·기준 단위의 타입화된 상태 추적(met/unmet/unknown/conflict/not_applicable), (2) 결측 변수의 **전역 중복 제거** 및 전역 질문 큐(세션당 최대 3라운드), (3) 답변 후 영향 기준만 **표적 재평가**, (4) 근거 문장 저장·충돌 시 인간 검토 라우팅으로 **재현 가능하고 감사 가능한** 판단.

## 2. 예상 API 사용량 및 USD 70 사용 계획

- **예상 요청 규모(합성 데이터 기준 추정)**: 환자 요약 20건(합성 10 + 교수 제공 10) × 시험 3개 × 기준 약 5개 ≈ **기준 매칭 300회/1회 실행**. 여기에 기준 파싱(시험당 1회), 프로파일 추출(환자당 1회), 질문 생성·답변 정규화·재평가(환자당 최대 3라운드), 결과 설명(환자·시험당 1회)을 더하면 **1회 전체 실행당 약 500~600 요청**, 개발 중 반복 실험(수십 회 실행 + 프롬프트 튜닝)을 포함해 **총 약 15,000~25,000 요청** 수준으로 추정.
- **모델 사용 분배(계획)**: 대량·정형 작업(기준 파싱, 프로파일 추출, 답변 정규화)은 **저비용 소형 모델**, 판단 품질이 중요한 작업(기준 매칭, 충돌 감지, 결과 설명)은 **상위 모델**로 이원화 — 예산의 약 60%를 소형 모델 대량 호출에, 40%를 상위 모델 핵심 호출에 배정.
- **프롬프트 캐싱**: 시험 프로토콜·기준 텍스트와 시스템 프롬프트는 환자가 바뀌어도 동일하므로 **프리픽스 캐싱**으로 반복 비용 절감(동일 시험을 여러 환자에 재사용하는 구조라 캐시 적중률이 높음).
- **데이터**: 개발·평가 전 과정에서 **합성/모의 데이터만 사용**(합성 환자 요약, 모의 시험 프로토콜, 라벨링된 매칭 시나리오, 교수 제공 입력 예시) — 실제 환자 데이터 없음.
- **비용 통제**: (1) 단계별 예산 상한(파싱/매칭/설명 각각 캡), (2) 이미 스키마에 정의된 **RequestLog**(요청별 지연·모델 버전·오류 기록)로 사용량 모니터링, (3) 배치 처리로 호출 수 축소, (4) 아키텍처가 강제하는 **질문 3라운드 상한**으로 대화 비용의 상한 보장, (5) 규칙 계층은 LLM 호출이 전혀 없어 판정 자체는 비용 0.

## 3. 기대 산출물 및 완성 목표

- **최종 산출물**: (1) 합성 시나리오를 끝까지 재현하는 실행 파이프라인(결정론적 매처 → LLM 에이전트 순차 교체), (2) 라벨링된 매칭 시나리오 기반 정확도 평가 리포트, (3) 질문 생성·재평가 데모, (4) 아키텍처·평가 문서 일체.
- **현재 진행 상황(사실 기준)**: 잠금된 v1.2-final 아키텍처의 전체 스키마 계층(Pydantic v2, 21개 모델·7개 열거형), 결정론적 규칙 계층, 10개 에이전트 타입 계약과 일부 휴리스틱 단계, 합성 데이터 검증 하네스(데이터셋 4종 + 검증 스크립트), 전체 워크플로 문서, **pytest 102개 전부 통과**(규칙·공유 상태·파싱·매칭·질문·답변 정규화·오프라인 데모 검증). 실제 LLM, RAG, LangGraph 런타임과 임상 성능 검증은 아직 구현하지 않음.
- **범위의 한계(명시)**: 본 시스템은 **연구용 프로토타입**으로 의학적 정확성을 주장하지 않으며, **실제 임상 의사결정 지원이나 의료 조언이 아님**. 모든 결과는 합성 데이터 기준 검증이며, 실환자 적용은 범위 밖임.
