# Evidence Table — Claims Mapped to Repository Files

Every project claim below is grounded in a file present in this
repository. The last column keeps the claims honest: it states what the
evidence does **not** establish.

| Claim | Evidence file | What it proves | What it does not prove |
|---|---|---|---|
| The architecture is not a simple chatbot | `models.py` (`PatientSession`, `CriterionState`, `FollowUpQuestion`); `docs/architecture.md`; `docs/state_transition.md` | The system is defined around a typed multi-trial session state with per-criterion five-value statuses, a bounded clarification loop, and review escalation — structures a chat transcript does not have | That the interactive loop runs end to end today (orchestration is not yet implemented) |
| Eligibility logic is separated from LLM reasoning | `rules.py`; `agents/criterion_matching.py` (docstring: effects are NOT decided here) | All eligibility effects and recommendations come from pure, deterministic functions; agent contracts and heuristic fallbacks produce match statuses and normalized inputs | That future LLM implementations will respect the boundary — that must be enforced in code review when agents are implemented |
| Multi-trial state is modeled | `models.py` (`trial_states_by_trial_id`, `TrialState`); `examples/demo_patient_session.json` | One session holds multiple trials, each with its own `trial_context` and `criterion_states`; the demo example instantiates two trials and validates | Scalability or behavior with real trial registries |
| Missing variables are globally deduplicated | `rules.deduplicate_missing_variables`; `tests/test_missing_variable_dedup.py`; `examples/demo_patient_session.json` (`ecog_performance_status` shared by both trials) | The same `missing_variable_key` across trials yields exactly one pool item, merging affected criterion and trial ids — verified by tests | That an LLM will name variables consistently enough to hit the same key (canonicalization is a TODO) |
| Professor patient summaries are handled as input examples, not ground truth | `examples/professor_patient_summaries.json` (read-only); `tests/test_professor_patient_summaries_load.py`; `docs/professor_patient_input_notes.md` | All 10 summaries validate against the dataset schema and pass through the extraction input contract; no test derives eligibility labels from them | Clinical extraction quality — the fallback parses only a small set of explicit fields |
| Core rules are covered by tests | `tests/test_effect_rules.py` (10 tests); `tests/test_recommendation_rules.py` (6 tests) | Every locked effect mapping, both review paths (conflict, max rounds), and all four precedence outcomes behave exactly as specified | Correctness of inputs to the rules (match statuses are assumed, not computed, in tests) |
| Relevance can never override hard eligibility | `rules.compute_trial_recommendation`; `tests/test_recommendation_rules.py::test_relevance_score_never_overrides_hard_block` | A trial with relevance 1.0 and one blocking criterion is still `likely_ineligible`; blocked trials always rank below non-blocked | Anything about how relevance scores are produced (estimation is future LLM work) |
| The clarification loop is bounded at 3 rounds | `models.py` (`clarification_round_count` with `le=3`, `MAX_CLARIFICATION_ROUNDS`); `rules.derive_eligibility_effect` (max-rounds path); `tests/test_effect_rules.py::test_unknown_after_max_rounds_requires_review` | The cap is enforced at the schema level and unknowns past the cap route to human review with a typed reason | That a runtime loop increments the counter correctly (tracker mutation is a stub) |
| The full schema composes into a valid session | `examples/demo_patient_session.json`; `tests/test_models_validate.py` | A realistic session — two trials, dedup'd missing variable, queued question, two recommendations — validates against `PatientSession` and respects the invariants | That the session was produced by running the system (it is a hand-built fixture) |
| The recommendation label space is fully exercised by fixtures | `examples/synthetic_matching_scenarios.json`; `tests/test_synthetic_matching_scenarios.py` | Six labeled scenarios cover all four `Recommendation` values with typed expected outputs | That one scenario-replay evaluator reproduces every label end to end; that evaluator is still planned |
| Trial ingestion is source-agnostic and offline | `models.TrialProtocol`; `examples/synthetic_trial_protocols.json`; `tests/test_synthetic_trial_protocols_load.py` | Protocols carry adapter-ready fields (`nct_id`, `source`, `source_url`, `retrieved_at`) as static data; nothing performs network calls | Compatibility with actual ClinicalTrials.gov API v2 responses (adapter not implemented) |
| The harness is reproducible by one command | `tests/` (14 files, 102 tests); `scripts/validate_synthetic_data.py`; `conftest.py` | `python -m pytest -q` passes 102/102 from the project root; the validator script validates all datasets and writes a summary from any cwd | Clinical accuracy and real LLM/API behavior are not tested |
