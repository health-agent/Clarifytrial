# ClarifyTrial 프로젝트 한눈에 보기

![ClarifyTrial 전체 워크플로](assets/clarifytrial-workflow.png)

## 이 프로젝트는 무엇인가

환자 한 명에게 여러 임상시험을 추천할 때, 현재 정보만으로 판단할 수 없는
조건을 찾아 질문하고, 답변을 받은 뒤 필요한 부분만 다시 판단하는 시스템이다.

```text
환자 문장 + 임상시험 기준
→ 기준별 최초 판단
→ 공통 부족정보를 질문
→ 환자 답변
→ 관련 기준만 재판정
→ 추천 순위와 이유 출력
```

## 가장 중요한 설계

### 1. 모든 판단을 한 상태에 저장한다

이전 데모는 각 에이전트의 JSON 출력을 다음 에이전트에 전달하는 방식이었다.
ClarifyTrial은 `PatientSession`에 다음 내용을 함께 저장한다.

- 구조화된 환자 정보
- 임상시험별 기준 판단
- 여러 시험이 공통으로 요구하는 부족정보
- 이미 물어본 질문과 환자 답변
- 답변으로 변경된 기준
- 최종 추천과 사람 검토 필요 여부

따라서 어떤 답변 때문에 어떤 임상시험의 판단이 바뀌었는지 추적할 수 있다.

### 2. 같은 질문은 한 번만 한다

여러 임상시험이 모두 ECOG, 임신 여부, 장기이식 이력처럼 같은 정보를
필요로 할 수 있다. 이를 trial별로 따로 질문하지 않고
`global_missing_variable_pool`에서 하나로 합친다.

예를 들어 임상시험 세 개가 ECOG를 요구해도 질문은 한 번만 하고, 그 답변을
관련된 세 시험에 모두 적용한다.

### 3. 답변과 관련된 기준만 다시 판단한다

환자가 ECOG에 답했다면 나이, 성별, 바이오마커까지 전부 다시 생성하지 않는다.
ECOG와 연결된 `criterion_id`만 찾아 재평가한다. 이것이 표적 재평가다.

### 4. LLM과 판정 규칙을 분리한다

LLM이 맡을 일:

- 긴 임상시험 기준 구조화
- 환자 문장에서 사실 추출
- 기준과 환자 근거의 의미 비교
- 질문을 환자가 이해하기 쉽게 표현
- 최종 설명 작성

Python 규칙이 맡을 일:

- 필수 선정기준 위반 처리
- 제외기준 충족 시 차단
- 불확실성과 근거 충돌 분리
- 질문 최대 3라운드 제한
- 관련성 점수가 hard block을 뒤집지 못하게 제한
- 추천 순서와 사람 검토 전환

## 이전 Health Agent 데모와 비교

| 항목 | 이전 6-call 데모 | ClarifyTrial 방향 |
|---|---|---|
| 실행 구조 | 정해진 6개 호출을 순서대로 실행 | 상태에 따라 질문·재판정·종료가 갈리는 그래프 |
| 상태 | 이전 에이전트 JSON을 다음 호출에 포함 | 중앙 `PatientSession`에 전체 변화 저장 |
| 질문 | trial별 unknown에서 질문 생성 | 여러 trial의 공통 부족정보를 합쳐 질문 |
| 환자 답변 | 질문 에이전트가 가상 답변도 생성 | 실제 사용자 입력 또는 독립된 숨은 환자 상태 사용 |
| 재판정 | 전체 최종 판단을 다시 생성 | 답변과 연결된 criterion만 재평가 |
| 최종 결정 | LLM 결과를 Python이 형식·일관성 보정 | 결정 규칙 자체를 Python에 고정 |
| 실패 처리 | 누락 JSON을 fallback으로 보충 | 충돌·반복 한도 초과를 사람 검토로 전환 |
| 연구 질문 | 주어진 후보를 얼마나 잘 판단하는가 | 어떤 질문이 최소 횟수로 추천을 가장 많이 개선하는가 |

## 구현 순서

### 1단계: 실행 껍질

- 기존 agent 함수를 LangGraph 노드로 감싼다.
- `PatientSession`을 그래프 상태로 사용한다.
- 질문이 필요하면 실행을 멈추고 답변 이후 재개한다.
- 기존 deterministic demo는 회귀 기준선으로 유지한다.

### 2단계: 데이터와 검색

- ClinicalTrials.gov API v2 adapter를 추가한다.
- 임상시험 기준 파싱 결과를 trial 단위로 캐시한다.
- BM25와 의학 임베딩을 결합한 후보 검색을 별도로 평가한다.
- 합성 환자의 정답 trial을 강제로 후보에 넣지 않는다.

### 3단계: 실제 LLM 연결

- Solar client를 공통 인터페이스 뒤에 연결한다.
- 기준 파싱, 환자 추출, 의미 매칭, 설명을 한 단계씩 교체한다.
- 각 교체 단계에서 deterministic 결과와 비교한다.
- LLM이 출력한 근거 문장과 criterion 원문 위치를 저장한다.

### 4단계: 질문 연구

- 질문 생성기와 환자 답변기를 분리한다.
- 숨은 전체 환자 상태에서 답변만 제공하는 평가기를 만든다.
- 여러 trial에서 공유되는 질문을 합친다.
- 판정을 바꿀 가능성이 큰 질문부터 최대 3~5개만 선택한다.
- 질문 전후 unknown 감소와 추천 변화가 올바른지 측정한다.

### 5단계: 대회 평가

- 검색, 기준별 판단, 질문, 답변 후 변화, 추천을 분리해 채점한다.
- 공개 개발 세트와 비공개 holdout을 분리한다.
- 처리 시간, API 실패, 실제 토큰 사용량, fallback 횟수를 함께 기록한다.
- 결과 화면에는 근거와 의료적 면책 고지를 항상 포함한다.

## 이 프로젝트에서 주장하지 않는 것

- 현재 휴리스틱이 임상적으로 정확하다고 주장하지 않는다.
- 합성 데이터 점수를 실제 환자 성능으로 해석하지 않는다.
- 에이전트 수가 많다는 것 자체를 연구 기여로 주장하지 않는다.
- LangGraph 도입 자체를 성능 향상으로 주장하지 않는다.

연구의 핵심은 **여러 임상시험의 부족정보를 통합하고, 가장 가치 있는 질문을
선택해, 답변과 관련된 판단만 안전하게 갱신하는가**이다.
