# ClarifyTrial 지원서 제출 양식

## 프로젝트명

**ClarifyTrial: 부족 정보 확인과 관련 기준 재평가를 포함한 임상시험 추천 시스템**

## 사용 대상

임상시험 코디네이터와 임상의

## 한 문장 소개

환자 정보와 임상시험 선정·제외 기준을 근거와 함께 비교하고, 판정에 필요한
정보가 부족하면 여러 후보 임상시험의 상태를 바꿀 수 있는 정보를 우선 확인한 뒤
관련 기준, 추천 순위와 설명을 갱신하는 연구용 대화형 임상시험 사전선별 시스템이다.

## 용어 표기

본문에서는 `criterion`을 **참여 기준**, `missing variable`을 **부족 정보**,
`targeted re-evaluation`을 **관련 기준 재평가**로 쓴다. `met`, `unmet`, `unknown`처럼
프로그램과 평가 자료가 사용하는 고정 판정값만 영문 코드로 남긴다.

## 문제 정의

임상시험 사전선별에서는 환자의 질환명만 비슷하다고 참여 가능하다고 판단할 수 없다.
연령, 병기, 바이오마커, ECOG, 이전 치료, 최근 검사값, 동반질환 등 여러 선정·제외
기준을 환자 기록과 하나씩 비교해야 한다. 그러나 초기 임상요약에는 이 정보가 모두
들어 있지 않은 경우가 많다. 기존의 한 번만 판단하는 방식은 부족한 값을 `unknown`으로
남긴 채 추천을 끝내고, 모든 정보를 묻는 방식은 질문과 조회 업무가 과도해진다.

ClarifyTrial은 `unknown`을 단순한 실패가 아니라 다음에 확인할 정보를 정하는 상태로
사용한다. 같은 정보가 여러 임상시험 기준에 필요하면 한 번만 확인하고, 새 답변과
연결된 참여 기준만 다시 평가한다.

## 연구 목표 및 접근 방법

1. 환자 임상요약, 임상 기록 또는 합성 FHIR 기록에서 표준 환자 정보를 추출하고 모든 값에 출처 문장을 연결한다.
2. ClinicalTrials.gov API에서 임상시험을 수집하고 참여 기준의 원문 위치, 수치·단위·시간 범위와 논리 구조를 보존해 저장한다.
3. BM25 단어 검색과 의미 유사도 검색을 결합해 후보 임상시험을 찾고, 검색 관련성과 실제 참여 가능성 판정을 분리한다.
4. 기준별 판정 에이전트가 각 참여 기준을 근거와 함께 `met`, `unmet`, `unknown`, `not_applicable`, `conflict`로 판정한다.
5. 부족 정보 관리기가 `unknown`의 원인을 찾고 여러 임상시험에 반복되는 정보를 하나의 공통 항목으로 합친다.
6. 다음 정보 선택기가 가능한 답에 따른 임상시험 상태 변화, 영향을 받는 임상시험·기준 수와 정보 획득 비용을 이용해 확인 순서를 정한다.
7. 핵심 실험에서는 구조화 전자의무기록, 임상 기록 검색, 직접 질문, 전문가 검토 순의 확인 절차를 고정하고 출처 선택 자체는 별도 합성 자료에서 평가한다.
8. 답변을 값·단위·시점·출처로 정규화한 뒤 그 정보와 연결된 참여 기준만 다시 평가한다.
9. 고정 조정 코드가 선정·제외 기준의 효과, 최종 참여 가능성과 추천 순위를 계산하고 대규모 언어모델은 구조화·근거·질문·설명 생성에 사용한다.
10. 부족 정보 확인 순서 실험과 전체·관련 기준 재평가 실험을 분리하고, TrialGPT 대화형 평가와 TREC 과거 검색 순위도 별도 표로 평가한다.

## 여러 에이전트의 역할

| 구성 | 역할 | 주요 출력 |
|---|---|---|
| 참여 기준 구조화 | 원문 선정·제외 기준을 원문 위치가 있는 논리 구조 트리로 변환 | 원문 항목, 최소 조건 ID, 논리, 필요한 환자 정보 |
| 환자 정보 이해 에이전트 | 환자 기록/FHIR에서 표준 정보와 근거 추출 | 환자 정보, 근거 문장 ID |
| 후보 임상시험 검색 | 구조화 필터, BM25, 의미 검색과 재정렬 | 후보 임상시험 목록 |
| 기준별 판정 에이전트 | 환자 근거와 각 참여 기준을 비교 | 상태, 이유, 근거, 부족 정보 |
| 부족 정보 관리기 | `unknown` 원인 추출, 여러 임상시험의 중복 제거 | 공통 부족 정보 목록 |
| 다음 정보 선택기 | 어떤 정보를 먼저 확인할지 고정 규칙으로 계산 | 선택한 정보와 우선순위 근거 |
| 정보 획득 관리기 | FHIR, 임상 기록, 직접 질문, 전문가 검토 순으로 확인 | 사용한 경로와 확인 결과 |
| 질문·답변 에이전트 | 정보 하나에 대한 질문 생성과 자유답변 정규화 | 질문 의도와 구조화된 답변 |
| 관련 기준 재평가 에이전트 | 답변과 연결된 참여 기준만 다시 판정 | 질문 전후 기준 상태 |
| 추천·설명 에이전트 | 참여 가능성, 추천 순위와 설명 생성 | 추천 목록, 판단 이유, 면책 고지 |

전체 상태는 환자별 `PatientSession`에 저장한다. 에이전트끼리 자연어 메시지만
주고받지 않고 참여 기준 ID, 부족 정보 키와 근거 ID가 있는 JSON으로 상태를
갱신한다. 언어모델이 낸 판정값은 형식 검사를 통과한 뒤 Python 고정 규칙으로
참여에 미치는 영향을 다시 계산한다.

## 단계별 구현 방법

### 1. 환자 정보 구조화

임상요약을 문장 ID로 나누고 연령, 성별, 질환, 병기, 바이오마커, ECOG,
치료명·치료 시점, 동반질환, 검사값과 지역을 추출한다. 수치에는 단위와 검사일,
모든 값에는 출처 문장 또는 FHIR 자원 ID를 붙인다. 기록에 없는 값은
없다고 추정하지 않고 `unknown`으로 둔다.

Synthea를 연결할 경우 `Patient`, `Condition`, `MedicationRequest`, `Procedure`,
`Observation`, `DiagnosticReport`를 초기 FHIR 자원으로 사용한다. 구조화 값이 있으면
직접 읽고, 없으면 임상 기록 검색 또는 코디네이터 질문으로 이동한다.

### 2. 임상시험 수집과 기준 구조화

ClinicalTrials.gov API v2에서 모집 상태, 질환과 지역을 필터링해 임상시험 기본 정보와
참여 기준을 가져온다. 기준의 목록 순서, 선정·제외 구역, 수치 범위, 단위, 시간 범위와
`A and (B or C)` 논리를 원문 위치가 있는 논리 구조 트리(Boolean AST)로 만든다.
원문 항목 하나가 여러 최소 조건으로 분리될 수 있으므로 개수 일치 대신 원문 포함
범위, 괄호 구조, 조건 ID와 수치 보존을 검증한다. 원본 응답을 해시값으로 고정하고
구조화 결과는 `NCT ID + 수정일 + 응답 해시 + 구조화 프로그램 버전`으로 저장한다.

선정 기준의 최소 조건은 충족해야 하는 조건으로, 제외 기준의 최소 조건은 해당하면
참여를 차단하는 조건으로 표준화하고 원문 위치를 함께 보존한다. 따라서 원문의
긍정·부정 표현과 관계없이 “제외 기준 충족은 참여 차단”이라는 의미가 일정하다.

### 3. 후보 검색

환자 질환, 하위 유형, 병기, 바이오마커, 치료 이력과 지역으로 검색어를 만든다.
BM25 단어 검색과 의미 유사도 검색을 함께 수행하고 RRF 방식으로 합친다. 후보 수는
고정하지 않고 TREC의 참여 가능 임상시험 재현율과 기준 판정 비용 곡선으로 정한다.
검색 점수는 관련성 순위에만 사용하며 명시적 제외 기준 위반을 뒤집지
않는다.

### 4. 기준별 판정

논리 구조 트리의 각 최소 조건이 요구하는 환자 정보와 근거를 연결한 뒤 수치, 단위,
시점과 부정을 비교하고, 전체 논리는 로컬의 참·거짓·확인 필요 규칙으로 합친다.
출력에는 모든 최소 조건과 참여 기준 ID, 판단, 근거 문장 ID, 이유와 부족 정보를
기록한다. 근거가 없으면 `unknown`, 서로 충돌하면 `conflict`로 둔다.

참여 기준 문장이 참인지 나타내는 판단과 실제 참여에 미치는 영향을 분리한다.
선정 기준의 `met`와 제외 기준의 `unmet`은 참여를 지원하고, 선정 기준의 `unmet`과
제외 기준의 `met`은 참여를 차단한다. 이 변환은 언어모델이 아니라 고정 규칙이
수행한다. 제출용 `satisfied`/`violated`에는 기준 종류와 `eligibility_effect`를
항상 함께 붙인다.

### 5. 다음 정보 획득 행동 선택

`unknown`인 참여 기준에서 부족 정보를 모은다. 의학 개념, 단위, 검체, 대상과 시간
범위까지 같을 때만 여러 임상시험의 항목을 하나로 합친다. 연속 수치는 참여 기준의
경계값에 따라 판정이 같은 구간으로 나누고 대표값만 대입한다. 초기 주 방식은 다음
우선순위 묶음을 위에서부터 차례대로 비교한다.

```text
priority_key(v) = (
  flip_capable_trial_count,
  affected_trial_count,
  affected_decision_critical_criterion_count,
  -minimum_feasible_route_cost_tier,
  stable_variable_key
)
```

- 가능한 값에 따라 상태가 바뀌는 임상시험 수를 가장 먼저 본다.
- 그다음 영향을 받는 임상시험 수와 차단 여부가 달라지는 참여 기준 수를 본다.
- 같은 경우 현재 확인 가능한 출처 정보에서 실행 가능한 획득 방법의 비용 등급을 비교한다.
- 마지막 고정 키는 같은 입력에서 같은 순서를 재현하기 위한 최종 정렬 기준이다.

가중 `heuristic_priority_score`는 우선순위 규칙 비교에만 사용하고 확률적 기대효용으로
부르지 않는다. 질문 횟수에는 기본 상한을 두지 않는다. 남은 정보가 임상시험 상태를
바꿀 수 없거나, 획득 가능한 정보가 없거나, 사용자가 더 답할 수 없을 때 종료한다.

### 6. 질문·답변과 관련 기준 재평가

선택한 정보에 구조화 전자의무기록 연결과 확인 가능한 출처 정보가 있으면 먼저
조회하고, 해결되지 않으면 임상 기록 검색, 그 뒤 직접 질문을 사용한다. 충돌하거나
전문 해석이 필요하면 사람 검토로 보낸다. 핵심 평가의 행동 한 번은 정보 하나와
질문 의도 하나만 다룬다. 질문 지시문에는 정보 정의, 허용 답변 형식, 단위, 시간
범위, 영향을 받는 참여 기준과 이미 물은 질문을 준다.

답변은 `value, unit, observed_at, source`로 정규화한다. 불명확한 답은 `unknown`을
유지하고 기존 정보와 충돌하는 답은 덮어쓰지 않는다. 정상 답변이면 기준 연결표에서
관련 참여 기준만 가져와 다시 평가하고 추천을 갱신한다.

### 7. 최종 추천과 설명

한 개라도 참여를 차단하는 기준이 있으면 `likely_ineligible`, 근거 충돌 또는 검토
대상이 있으면 `needs_human_review`, 차단은 없지만 판정에 중요한 기준 중
`unknown`이 하나라도 있으면 `uncertain`이다. 적용 가능한 모든 선정 기준이 `met`이고
모든 제외 기준이 `unmet`일 때만 `likely_eligible`이다.
추천 순위는 참여 가능성, 임상시험 관련성과 남은 불확실성을 함께 사용하되
명시적으로 차단된 임상시험은 뒤에 둔다. 설명 에이전트는 이미 결정된 상태에서 지원 기준,
차단 기준, 남은 부족 정보, 질문·답변과 순위 이유를 읽기 쉽게 정리한다.

## 데이터와 정답 자료

| 평가 구분 | 데이터와 사용 방법 |
|---|---|
| 임상시험 입력 | [ClinicalTrials.gov](https://clinicaltrials.gov/data-api/about-api)의 시점을 고정한 임상시험 기본 정보와 참여 기준 |
| 기준 구조화 | [Leaf Clinical Trials Corpus](https://www.nature.com/articles/s41597-022-01521-0)의 의학 개념·원문 위치·속성 주석 |
| A. 기준별 판정 | [TrialGPT Criterion Annotations](https://huggingface.co/datasets/ncbi/TrialGPT-Criterion-Annotations)의 전문가 기준 판정과 근거 문장 |
| B. 대화형 정보 확인 | TrialGPT에서 파생한 복수 부족 정보 자료의 질문 순서, 답변 반영과 종료 평가 |
| C. 후보 검색 | [TREC Clinical Trials 2021](https://trec.nist.gov/data/trials2021.html)·[2022](https://trec.nist.gov/data/trials2022.html)의 과거 환자 주제·정답 검색 순위 평가 |
| D. 정보 출처 선택 | [Synthea](https://github.com/synthetichealth/synthea) 등을 이용해 동기화한 합성 FHIR·임상 기록·직접 답변 사례 |

네 평가 구분은 환자와 임상시험 정답 체계가 다르므로 점수를 합산하지 않는다. 특히
TREC 정답은 대화형 비공개 정답지에 넣지 않고 과거 후보 검색 결과로만
보고한다.

### 불완전 환자 기록 생성

TrialGPT 주석에서 전문가 근거가 있는 환자-참여기준 쌍을 선택하고, 같은 환자의
여러 임상시험을 한 평가 사례로 묶는다. 질문 순서를 평가하는 사례에는 서로 다른
부족 정보를 여러 개 지운다. 같은 사실을 표현하는 모든 문구를 자연스럽게 삭제하거나
다시 쓰고 `[MASK]` 같은 위치 표시는 사용하지 않는다. 완전한 기록, 답을 얻을 수
없는 `unknown`, 추가 행동이 필요 없는 사례도 둔다.

정보를 지운 상태의 정답은 `unknown`, 원래 구조화된 값을 답한 뒤 정답은 원래 전문가
판정으로 둔다. 참여 기준을 반대로 판정하게 하는 합성 답변은 기준 경계값의 유효한
구간에서 만들고 다른 환자 사실과의 일관성을 확인한다. “모름” 또는 문맥으로 해석할
수 없는 답은 `unknown` 유지 분기로 둔다. 질문은 문장 전체가 똑같은지가 아니라
요구 정보, 단위, 시간 범위가 함께 맞는지 평가하고, 질문 의도가 맞을 때만 비공개
답변을 반환한다.

각 사례는 `완전한 기록 → 전문가 판정`, `일부 정보를 지운 기록 → unknown`,
`원래 답변 반영 → 전문가 판정 복원`, `무관한 참여 기준 → 변화 없음` 조건을
통과해야 한다.

### 비공개 정답지 구성

- `CriterionBenchmarkCase`: TrialGPT 참여 기준 판정과 근거 정답
- `InteractiveAcquisitionSession`: 원본·불완전 환자 기록, 복수 부족 정보,
  질문 의도, 구조화 답변 분기, 전후 참여 기준과 고정 규칙에서 계산한 임상시험 상태
- `TrecRetrievalTopic`: 과거 환자 주제, 자료 버전과 검색 순위 정답
- `SyntheticRouteCase`: 같은 사실의 출처별 존재 여부와 기대 정보 획득 경로

실행 에이전트에는 공개 입력만 주고 비공개 정답은 별도 파일과 채점 과정에서만
읽는다. 대화형 임상시험 상태는 완전한 환자 기록과 고정 규칙에서 계산한 값으로
표시하며 TREC 정답이나 임상 전문가 정답이라고 부르지 않는다.

## 비교 실험

### 실험 A: 부족 정보를 확인하는 순서

| 정책 | 동작 |
|---|---|
| 추가 확인 없음 (`No-acquisition`) | 추가 정보를 얻지 않고 현재 상태로 종료 |
| 고정 순서 (`Fixed-order`) | 원문 순서 또는 고정 정보 키 순서로 확인 |
| 연결 기준 수 우선 (`Coverage-only`) | 연결된 참여 기준 수가 많은 정보부터 확인 |
| 판정 변화 우선 (`Clarify-priority`) | 임상시험 상태 변화 가능성을 우선한 순서 |
| 모든 정보 확인 (`Ask-all`) | 답을 얻을 수 있는 모든 정보를 확인 |

모든 방식에 같은 시작 상태, 부족 정보 목록, 비공개 답변, 기준 판정기와 재평가
프로그램을 사용한다. 순서만 평가할 때는 정답 부족 정보 목록을 직접 주는 모드와,
전체 흐름에서 부족 정보 탐지 결과부터 쓰는 모드를 나눈다.

### 실험 B: 다시 판정하는 기준의 범위

| 방식 | 동작 |
|---|---|
| 전체 재평가 | 같은 답변 뒤 모든 후보 임상시험의 모든 참여 기준을 다시 평가 |
| 관련 기준 재평가 | 같은 답변 뒤 기준 연결표로 연결된 참여 기준만 평가 |

두 방식에는 같은 정보 확인 순서와 구조화 답변 이력을 다시 넣는다. 결과 상태
일치도와 무관한 기준의 변경 여부, 호출·토큰·비용·응답 시간을 측정한다. 따라서
정보 확인 순서의 효과와 관련 기준 재평가의 계산 효과가 한 비교에 섞이지 않는다.

## 평가 방법

| 평가 대상 | 지표 |
|---|---|
| 참여 기준 판정 | 선정·제외 기준별 F1, 전체 균형 F1, 오분류표 |
| 근거 연결 | 근거 문장 정밀도·재현율·F1, 근거 없는 주장 비율 |
| 부족 정보 탐지 | 가린 정보 재현율, 잘못 찾은 정보 비율, MRR |
| 질문 내용 | 요구 정보·단위·시간 범위·전체 의도 일치율 |
| 답변 정규화 | 구조화 값 정확도, 충돌 탐지, 잘못 확정한 비율 |
| 대화형 결과 | 기준 정답 회복률, 임상시험 상태 일치율, 무관한 기준 변경률 |
| 확인 순서 효율 | 행동 수·비용에 따른 품질 곡선 아래 면적, 목표 품질 도달 비용 |
| 종료 판단 | 너무 일찍 끝낸 비율, 아직 판정을 바꿀 수 있는 정보 수 |
| TREC 후보 검색 | nDCG@10, P@10, RPrec, MRR; 참여 가능 임상시험 Recall@k 보조 |
| 재평가 효율 | 전체·관련 기준 재평가 결과 일치율, 호출·토큰·비용·응답 시간 |

TREC 평가는 공식 정답과 같이 eligible=2, excluded=1, not relevant=0의 nDCG 가중치를
사용한다. unknown 결과는 `correctly_resolved`, `wrongly_committed`,
`remaining_unknown`으로 나눈다. 확인 행동이 증가할 때 완전한 정보 기준의 정답
회복률과 누적 비용을 매 단계 기록해 적절한 종료 구간을 찾는다. 원래 값 복원·반대
조건·불명확 답변은 먼저 각각 보고한다.

## 세부 검증 실험

- 임상시험 사이의 부족 정보 중복 제거를 끄고 임상시험마다 질문했을 때 반복 질문 수 비교
- 판정 변화 우선 방식을 고정 순서 또는 연결 기준 수 우선으로 바꿨을 때 품질·비용 곡선 비교
- 판정 변화 가능성을 빼고 연결된 기준 수만 사용했을 때 비교
- 같은 답변 이력에서 관련 기준 재평가와 전체 재평가의 상태·호출 수 비교
- 근거 문장을 요구한 판정기와 판정값만 요구한 판정기의 근거 F1 비교
- 종료 판단 없이 답을 얻을 수 있는 모든 정보를 확인했을 때 비교

정보 출처 선택 절차는 별도 `SyntheticRouteCase`에서 평가한다. 각 비교는 같은 환자에
대한 두 결과를 짝지어 만들고 부트스트랩으로 차이의 95% 구간을 계산한다.

TrialGPT 파생 자료는 환자 ID 단위로 개발용과 최종 평가용을 나누며, 같은 환자의
임상시험, 정보 삭제 변형과 답변 분기는 같은 구분에 둔다. 우선순위의 가중치·정규화·
종료 조건은 개발용 자료에서 고정하고 최종 평가에서는 바꾸지 않는다. TREC 결과는
공식 지표로 별도 산출하며 대화형 점수와 합산하지 않는다.

### 핵심 논문 해석

**TrialGPT.** 자유형식 환자 기록에서 후보 임상시험을 찾고, 각 선정·제외 기준을
근거 문장과 함께 판정한 뒤 추천 순위를 만든다. 공개된 1,015개의 전문가 검토
환자-참여기준 쌍은 ClarifyTrial의 최초 기준 판정과 근거 연결을 평가하는 직접 자료다.
다만 고정된 환자 기록을 사용하므로 부족 정보를 질문하고 답변 뒤 결과를 바꾸는
과정은 평가하지 않는다.

**TrialMatchAI.** 의학 용어 표준화, BM25 단어 검색과 의미 검색의 결합, 참여 기준
재정렬과 구조화 설명을 한 흐름으로 구현했다. ClarifyTrial은 이 검색 설계를
가져오되, 검색 관련성 점수로 명시적인 제외 기준 위반을 뒤집지 않는다.

**Learning to Ask Medical Questions와 FollowupQ.** 전자는 가려진 환자 정보 중
다음에 확인할 항목을 환자별로 다르게 고르는 문제를 다뤘고, 후자는 환자 메시지와
진료 기록을 함께 사용해 후속 질문을 만들고 같은 정보를 요구하는지 평가했다.
ClarifyTrial은 이를 여러 임상시험의 공통 부족 정보 선택과 질문 의도 채점에 적용한다.

**Chen 등(2025).** 참여 기준을 환자용 문진표로 바꾸고, 답변을 기준과 연결해 참여
가능성을 평가했다. 따라서 “참여 기준에서 후속 질문을 만든다”는 기능 자체는 이미
존재한다. ClarifyTrial은 임상시험별 전체 문진표를 먼저 묻는 대신, 여러 임상시험의
판정을 가장 크게 바꿀 정보를 순서대로 확인한다.

**Yang(2026).** 비슷한 참여 기준 묶기, 환자용 질문, 답변 판정과 대화 중 임상시험
제거를 결합한 가장 가까운 직접 선행연구다. ClarifyTrial은 이 연구와 비교해 공통
부족 정보를 합치는 정확한 조건, 판정 변화 가능성에 따른 확인 순서, 관련 기준만
재평가했을 때 전체 재평가 결과가 유지되는지를 각각 분리해 측정한다.

### ClarifyTrial이 검증할 부분

1. 의학 개념, 단위, 검체, 대상과 시간 범위가 같은 부족 정보를 여러 임상시험에서
   하나로 합치면 반복 질문이 줄어드는가?
2. 가능한 답에 따라 참여 가능성이 바뀔 수 있는 정보를 먼저 확인하면 고정 순서보다
   적은 행동으로 완전한 정보의 판단에 가까워지는가?
3. 같은 답변 이력에서 관련 참여 기준만 다시 판정해도 전체 재평가와 같은 결과를
   유지하면서 호출 수·응답 시간·비용이 줄어드는가?
4. 여러 부족 정보, 질문 의도, 구조화 답변, 질문 전후 판정을 비공개 정답지로 함께
   평가할 수 있는가?

후보 검색, 기준별 판정과 후속 질문 자체를 새 기능으로 주장하지 않는다. 위 내용은
현재 확인한 선행연구를 바탕으로 세운 비교 가능한 연구 가설이며, “최초” 또는
“유일”이라는 표현을 사용하지 않는다.

## 예상 API 사용량 및 비용 계획

LLM API는 참여 기준 구조화, 환자 정보 추출, 기준별 판정, 질문 문장,
자유답변 정규화와 최종 설명에 사용한다. 후보 필터링, 검색 결과 통합,
부족 정보 통합, 우선순위 계산, 상태 갱신, 참여 영향 변환, 재평가 대상 선택과
지표 계산은 로컬 코드로 처리한다.

환자 한 명의 전체 과정에 필요한 호출 수는 다음 식으로 측정한다.

```text
patient extraction
+ uncached trial parsing
+ initial matching batches
+ 정보 획득 횟수 * (question + answer normalization + linked rematching)
+ final explanation
```

임상시험 참여 기준 구조화 결과와 의미 표현은 `NCT ID + 수정일 + 응답 해시 + 구조화
프로그램 버전`으로 저장해 재사용하고, 한 임상시험의 여러 기준은 묶어서 판정한다.
답변 뒤에는 연결된 참여 기준만 다시 호출하며 구조화 실패 항목만 재시도한다. 각
요청의 입력·재사용 입력·출력 토큰, 응답 시간, 재시도와 실행 시점 단가를 저장해
환자 한 명당 비용을 계산한다.

```text
uncached_input_tokens = total_input_tokens - cached_input_tokens

cost = uncached_input_tokens * uncached_input_rate
     + cached_input_tokens   * cached_input_rate
     + output_tokens         * output_rate
```

저장 결과가 없는 최초 실행과 저장 결과를 재사용한 실행의 비용을 분리해 재사용
토큰을 중복 계산하지 않는다.

예산을 임의 금액으로 고정하지 않는다. 소규모 예비 실행에서 단계별 토큰 비중과
응답 시간을 확인한 뒤 후보 수, 한 번에 판정할 기준 수와 평가 규모를 정한다.
모델별 비교는 정확도뿐 아니라 참여 기준 하나와 환자 한 명의 전체 과정을 끝내는
비용을 함께 보고한다.

## 현재 구현과 제안 구현

현재 저장소에는 Pydantic 기반 `PatientSession`, 선정·제외 기준의 참여 영향 규칙,
합성 환자·임상시험 입력용 로컬 기준 구조화·판정 규칙, 공통 부족 정보 중복 제거,
질문 틀, 단일 정보 답변 정규화와 추천 골격이 구현되어 있다.

본 제안에서 구현·검증할 범위는 ClinicalTrials.gov·TrialGPT·TREC 연동 모듈, 참여
기준 논리 구조 트리, 실제 LLM 기준 판정기와 환자 근거 검색, 부족 정보 연결표와
우선순위 규칙, 질문 대기열과 연결된 관련 기준 재평가, 네 가지 평가 자료형과 공개
평가 실행기다. Synthea FHIR와 임상 기록 조회는 핵심 결과와 분리한 선택 시연으로 둔다.

## 기대 산출물과 초기 구현 범위

### 필수 산출물

- ClinicalTrials.gov 임상시험 연동 모듈과 참여 기준 저장소
- TrialGPT 기준별 판정 평가와 TREC 과거 검색 순위 평가 실행기
- 여러 정보를 지우는 생성기, 공개 입력 묶음과 채점기 전용 비공개 정답지
- 환자 입력부터 질문 전·후 판정, 추천과 설명까지 이어지는 멀티에이전트 실행기
- 다섯 가지 정보 확인 순서 비교와 전체·관련 기준 재평가 결과
- 기준별 근거, 부족 정보, 질문·답변과 전후 상태가 담긴 결과 JSON
- 행동 수별 기준 정답 회복률·비용 곡선과 환자별 API 사용량
- 재현 명령, 데이터 출처와 의료적 면책 고지

### 선택 기능

Synthea FHIR 조회와 임상 기록 검색을 연결해 구조화 전자의무기록, 임상 기록,
코디네이터 질문과 전문가 검토를 차례로 거치는 과정을 시연한다.

## 최종 데모

1. 합성 환자 임상요약 또는 합성 FHIR 기록을 입력한다.
2. 관련 ClinicalTrials.gov 임상시험과 각 참여 기준을 보여준다.
3. 최초 기준 판정, 근거와 `unknown` 원인을 보여준다.
4. 다음으로 확인할 정보와 선택한 정보 획득 방법을 보여준다.
5. 숨겨둔 환자 답변을 넣고 연결된 참여 기준만 바뀌는 과정을 보여준다.
6. 질문 전·후 참여 가능성, 추천 순위, 근거, 남은 불확실성과 API 비용을 비교한다.
7. 고정 순서·연결 기준 수 우선·판정 변화 우선 방식의 품질·비용 곡선과 모든 정보
   확인 결과를 제시하고, 전체·관련 기준 재평가 결과는 별도 표로 보여준다.

## 최종 출력

최종 결과에는 제출용 `eligible`, `ineligible`, `uncertain`과 내부
`screening_status`를 함께 둔다. 참여 기준별 `satisfied`, `violated`, `unknown`,
`not_applicable`에는 `criterion_type`, `criterion_truth`, `eligibility_effect`를
붙여 제외 기준의 `satisfied`가 참여 차단일 수 있음을 명확히 한다.
근거 문장, 부족 정보 질문과 답변, 질문 전·후 변화, 환자별 추천 순위와 설명,
`review_required` 사유와 의료적 면책 고지를 포함한다.

## 의료적 면책

ClarifyTrial은 합성 환자와 공개 임상시험 데이터에서 평가되는 연구용 사전검토
프로토타입이다. 출력은 의료적 자문, 임상시험 적격성 확정 또는 등록 결정을
대체하지 않으며, 최종 판단은 자격을 갖춘 임상 전문가가 수행해야 한다.
